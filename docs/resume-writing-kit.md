# 简历写作作战手册

这份文档用于把仓库中的履历内容、项目证据和量化口径整理成一套可复用的简历素材。目标不是再写一版经历，而是让后续你在写不同版本简历时，能快速判断“该选什么、怎么写、哪些数字能用、哪些只能保守表述”。

## 文档地图

- `docs/resume-experience-copy.md`
  直接可复制到 Word / PDF 简历中的经历文案。
- `docs/resume-star-bank.md`
  面向 STAR 结构的项目/经历证据库，适合写项目详情、准备面试追问、扩写自我介绍。
- `docs/deployment-channels.md`
  用于补充 GitHub Pages / Vercel / 自托管部署背景，在被问到工程化交付时可直接引用。

## 当前定位建议

优先定位：
- AI 应用工程师（RAG / Agent）
- LLM 应用工程师 / 智能体工程师
- 后端工程师（偏 AI 工程、平台工程、业务系统集成）

可兼容定位：
- 全栈工程师（需保留 EasyCloudPan、乐学网、作品集网站等前后端交付内容）
- 平台工程 / 工程效率方向（需强调部署、回归门禁、可观测性、质量基线）

不建议作为主投递方向：
- 纯算法研究岗
- 纯前端视觉岗

## 可保守引用但不宜过度扩大的结果

以下内容适合写成“形成能力”“支撑场景”“建立机制”，不建议擅自补百分比或用户规模：

- 南方科技大学 IPA 原型交付与验收
- 国家骨科临床研究中心小程序后端
- 乐学网课程/支付/互动闭环
- 九州通四向穿梭车路径规划系统的仿真与调度验证

推荐写法：
- 按期交付原型并通过验收
- 支撑核心业务流程稳定运行
- 完成关键算法验证与可视化演示
- 建立可复现、可调试的工程闭环

## Bullet 写法约束

推荐结构：
- 动词 + 负责对象 + 技术动作 + 结果 / 影响

推荐动词：
- 设计
- 重构
- 构建
- 实现
- 推动
- 优化
- 建立
- 落地
- 治理
- 打通

不推荐写法：
- 参与了
- 做了
- 帮助团队
- 负责一些
- 主要是

## 面试准备优先级

优先准备成 3 分钟完整讲述的项目：
1. 智能客服运行时
2. 微信智能助手
3. RAG-QA System
4. EasyCloudPan
5. 联通数据中台优化

其中至少要有两类讲法：
- 一类偏 AI 工程：检索、路由、Agent、可恢复工作流、知识治理
- 一类偏后端工程：性能、安全、缓存、数据迁移、交付与监控

## 使用建议

写简历时按以下顺序操作：
1. 先从 `docs/resume-experience-copy.md` 复制基础文案。
2. 再到 `docs/resume-star-bank.md` 给重点项目补背景、难点或结果细节。
3. 最后对照本文件做岗位取舍，删除与目标岗位弱相关的项目。

如果需要写英文简历，优先翻译“职责 + 技术动作 + 结果”三段式，不要逐字翻译中文修饰语。
