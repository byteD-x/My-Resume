# 简历补数清单

> 用途：给 `docs/resume-ai-main.md` 补齐可以显著增强说服力、但当前仓库与公开材料尚未完整验证的量化指标。所有数据建议保留“时间窗 + 口径 + 来源”。

## 使用原则

- 只补你能拿出截图、日志、监控、导出报表、后台统计或 GitHub 页面证明的数据。
- 每个数字最好同时记录：
  - 指标名
  - 时间窗
  - 起点 / 终点
  - 数据来源
  - 是否可对外公开
- 如果只能拿到局部数据，优先写区间、峰值或阶段性结果，不要硬凑全年口径。

## 论文检索任务平台

### 优先补齐

- 用户规模：注册用户数、活跃用户数、试用用户数 `[补充数据]`
- 任务规模：累计任务数、日均任务数、峰值并发任务数 `[补充数据]`
- 支付结果：付费用户数、支付转化率、客单价、Pro 升级订单量 `[补充数据]`
- 稳定性：平均任务耗时、SSE 同步成功率、轮询兜底触发比例 `[补充数据]`
- 质量验证：pytest / Playwright / 真实 E2E 用例数与通过率 `[补充数据]`

### 可写进简历的位置

- 项目结果句：
  - 完成测试验收版交付，覆盖 `[补充数据]` 个核心用户主路径与 `[补充数据]` 个管理端运营动作。
- 技术结果句：
  - 通过 `pytest / Vitest / Playwright / 真实 E2E` 建立验证基线，累计覆盖 `[补充数据]` 条关键测试场景。
- 业务结果句：
  - 上线试运行阶段支撑 `[补充数据]` 次论文检索任务与 `[补充数据]` 次支付升级。

### 建议取数来源

- 后端日志 / 指标
- 任务表、订单表、额度流水表导出
- Playwright / pytest 报告
- 管理后台统计截图

## RentBox 共享擦窗宝小程序

### 优先补齐

- 上线规模：柜机数、点位数、城市数、渠道数 `[补充数据]`
- 交易规模：累计订单量、日均订单量、支付成功率、退款率 `[补充数据]`
- 设备表现：开锁成功率、异常工单量、设备在线率 `[补充数据]`
- 免押链路：支付分授权成功率、免押转化率 `[补充数据]`
- 交付结果：正式上线日期、发布次数、回滚演练次数 `[补充数据]`

### 可写进简历的位置

- 项目结果句：
  - 完成共享擦窗宝小程序正式环境发布，支撑 `[补充数据]` 台柜机 / `[补充数据]` 个点位的上线运营口径。
- 稳定性结果句：
  - 收口支付、开锁、审计与导出高风险链路后，将关键故障项压缩至 `[补充数据]`。
- 多端交付句：
  - 推动管理端、渠道端与小程序持续去 mock 并接真实后端，累计打通 `[补充数据]` 条核心业务链路。

### 建议取数来源

- 管理端后台统计
- 审计日志 / 订单导出
- 支付平台对账单
- 设备平台日志
- 正式环境发布记录与健康检查截图

## 通用建议

- 简历里能直接写死的数字，优先使用已验证项：
  - `20s+ -> 4s`
  - `300+` 表
  - `3 亿+` 记录
  - `5x`
  - `1000+` 并发上传
- 最近项目如果没有可公开指标，不要硬写模糊“大幅提升”“显著增长”，改为：
  - 已完成测试验收版交付
  - 已完成正式环境发布与健康校验
  - 已打通主链路并建立自动化验证基线
